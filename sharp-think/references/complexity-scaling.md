# 复杂度分级与思考预算

> 当遇到分级边界案例，或需要更细致地判断复杂度时，参考此文档。

## 详细分级标准

### L1 直接回答

**判定条件**（满足任意一条）：
- 答案可以直接从已有知识中获取
- 只需要单步操作（翻译、格式转换、简单计算）
- 用户明确说"简单回答"或"快一点"
- 问题有唯一正确答案（事实性问题）

**思考预算**：
- 不展开推理过程
- 直接给出答案
- 如果不确定，标注"我不确定"而非猜测

**常见误判**：
- ❌ "帮我写一封邮件" → 不是 L1（需要理解上下文和意图）
- ✅ "当前北京时间" → 是 L1

---

### L2 标准分析

**判定条件**（满足任意两条）：
- 需要 2-5 步推理才能得出答案
- 涉及 2-4 个因素的综合考虑
- 有明确的约束条件需要满足
- 需要一定的专业判断，但不需要多方案深度对比
- 输出需要一定结构，但不需要完整报告

**思考预算**：
- 使用 L2 推理模板（目标 → 约束 → 方案 → 验证）
- 推理步骤控制在 4 步以内
- 方案探索阶段考虑 1-2 个选项即可
- 不需要引用参考卡片（除非遇到认知偏差风险）

**常见误判**：
- ❌ "帮我设计一个完整的营销方案" → 不是 L2（需要多方案对比、风险评估 → L3）
- ✅ "帮我写一份项目周报" → 是 L2（有模板可循，不需要深度分析）

---

### L3 深度分析

**判定条件**（满足任意两条）：
- 需要 5 步以上的推理链
- 涉及多方案对比和权衡
- 影响范围大、决策成本高（一旦选错代价大）
- 涉及多个利益相关方
- 需要跨领域知识综合
- 用户明确要求"深入分析"、"详细方案"、"全面评估"

**思考预算**：
- 使用 L3 完整推理模板
- 方案探索至少 2 个选项
- 引用相关参考卡片
- 进行认知偏差自检
- 通过完整质量检查清单

**常见误判**：
- ❌ "这个函数为什么报错" → 通常不是 L3（除非是极其复杂的系统级 bug）
- ✅ "我们应该用 React 还是 Vue 重构前端" → 是 L3（多方案对比、影响范围大）

---

## 边界案例处理

以下情况建议升级到更高级别：

| 情况 | 建议 |
|------|------|
| 不确定是 L1 还是 L2 | 按 L2 处理（多花 10 秒比漏掉重要因素好） |
| 不确定是 L2 还是 L3 | 按 L3 处理（宁可多分析一步） |
| 用户似乎问了一个简单问题，但直觉告诉你背后有更复杂的需求 | 先按用户字面意思回答（L1/L2），然后追问"是否需要更深入的分析？" |
| 任务可以拆分为多个独立子问题 | 对每个子问题独立分级，但整体框架按最高级别处理 |

---

## 思考预算参考

| 级别 | 推理步骤上限 | 方案探索数量 | 是否引用参考卡片 | 质量检查严格度 |
|------|------------|------------|----------------|--------------|
| L1 | 0（直接回答） | 0 | 否 | 基本正确即可 |
| L2 | 4 步 | 1-2 个 | 仅在需要时 | 标准检查 |
| L3 | 无上限 | 2+ 个 | 推荐 | 严格检查 |

---

## 动态调整

思考过程中如果发现：
- **复杂度被低估了**：立即升级，不要在低级别框架中强行处理
- **复杂度被高估了**：可以降级，但要在输出中说明"这个问题比预期简单，核心原因是..."
- **用户追加需求导致升级**：从当前步骤继续，不需要从头开始，但需要补充新的分析维度
